## 2. 역할 기반 접근 제어(RBAC) 설계

### 2.1 일반 기업의 직급/직무별 권한 체계

한국 기업의 전통적 직급 체계:

```
대표이사(CEO) → 임원(CxO/이사) → 부장 → 차장 → 과장 → 대리 → 주임 → 사원
```

RBAC의 핵심 원칙:
- **최소 권한 원칙(Least Privilege)**: 업무에 필요한 최소한의 권한만 부여
- **역할 기반 할당**: 권한을 사용자에게 직접 부여하지 않고 역할(Role)에 부여한 후, 사용자를 역할에 할당
- **자동 프로비저닝**: 직급/부서 변경 시 권한이 자동으로 갱신 (예: 마케팅 → 개발 이동 시 마케팅 권한 자동 해제, 개발 권한 자동 부여)

### 2.2 버스 회사 특화 역할 체계

```
대표이사 (OWNER)
  └── 관리자/배차담당 (ADMIN)
        └── 운행관리자 (MANAGER)
              └── 버스기사 (DRIVER)
```

#### 역할별 상세 정의

| 역할 | 코드 | 설명 | 주요 책임 |
|------|------|------|----------|
| 대표이사 | `OWNER` | 회사 최고 의사결정자 | 경영 현황 총괄, 최종 승인, 통계/리포트 조회 |
| 관리자 | `ADMIN` | 배차 담당, 인사/총무 | 배차표 생성/수정, 기사 관리, 급여 정산, 결재 처리 |
| 운행관리자 | `MANAGER` | 현장 운행 감독 | 당일 운행 모니터링, 돌발 상황 대응, 기사 출근 확인 |
| 기사 | `DRIVER` | 버스 운전 실무 | 배차 확인, 출퇴근 기록, 휴무/교대 요청, 사고 보고 |

### 2.3 기능별 접근 권한 매트릭스

| 기능 | OWNER | ADMIN | MANAGER | DRIVER |
|------|:-----:|:-----:|:-------:|:------:|
| **대시보드(경영현황)** | R | R | - | - |
| **대시보드(운행현황)** | R | R | R | - |
| **배차표 생성/수정** | R | CRUD | R | - |
| **배차표 조회(전체)** | R | R | R | - |
| **배차표 조회(본인)** | - | - | - | R |
| **기사 정보 관리** | R | CRUD | R | - |
| **기사 본인 정보** | - | - | - | RU |
| **급여 관리(전체)** | R | CRUD | - | - |
| **급여 조회(본인)** | - | - | - | R |
| **전자결재(기안)** | C | C | C | C |
| **전자결재(승인/반려)** | CRUD | CRUD | RU | - |
| **휴무 요청** | - | - | C | C |
| **휴무 승인** | RU | RU | - | - |
| **차량 관리** | R | CRUD | R | R |
| **사고 보고(작성)** | - | - | C | C |
| **사고 보고(조회/처리)** | R | CRUD | R | - |
| **공지사항 작성** | C | C | C | - |
| **공지사항 조회** | R | R | R | R |
| **시스템 설정** | CRUD | R | - | - |
| **감사 로그** | R | R | - | - |

> C=Create, R=Read, U=Update, D=Delete

### 2.4 데이터 가시성 규칙

```
[상위자 → 하위자 데이터 전체 조회 가능]
OWNER: 전사 데이터 조회
ADMIN: 전체 기사/운행/차량 데이터 조회
MANAGER: 담당 노선/팀 기사 데이터 조회
DRIVER: 본인 데이터만 조회

[민감 데이터 접근 제한]
- 급여 상세: OWNER, ADMIN만 전체 조회 / DRIVER는 본인 것만
- 인사 평가: OWNER, ADMIN만 조회
- 시스템 로그: OWNER만 접근
```

### 2.5 기술적 구현 패턴

```typescript
// Prisma 스키마 예시
enum Role {
  OWNER
  ADMIN
  MANAGER
  DRIVER
}

model User {
  id        String   @id @default(uuid())
  role      Role     @default(DRIVER)
  // ...
}

// 미들웨어 예시
function authorize(...allowedRoles: Role[]) {
  return (req, res, next) => {
    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

// 데이터 가시성 필터
function getDataFilter(user: User) {
  switch (user.role) {
    case 'OWNER':
    case 'ADMIN':
      return {}; // 전체 데이터
    case 'MANAGER':
      return { routeId: { in: user.managedRouteIds } };
    case 'DRIVER':
      return { driverId: user.id }; // 본인 데이터만
  }
}
```

## 3. 핵심 모듈별 상세 기능

### 3.1 전자결재 시스템

#### 워크플로우 설계

```
기안자(작성) → 검토자(검토) → 결재자(승인/반려) → 최종결재자(최종승인)
     ↓              ↓                ↓
  [임시저장]     [의견첨부]     [반려 시 기안자에게 반환]
                              [대결(부재 시 대리 결재)]
```

#### 결재 상태 관리

| 상태 | 설명 | 알림 대상 |
|------|------|----------|
| `DRAFT` | 임시 저장 | - |
| `PENDING` | 결재 대기 | 다음 결재자 |
| `IN_REVIEW` | 검토 중 | - |
| `APPROVED` | 승인 완료 | 기안자 |
| `REJECTED` | 반려 | 기안자 |
| `CANCELLED` | 기안 취소 | 결재선 전체 |

#### 버스 회사 특화 결재 양식

- **휴무 신청서**: 기사 → 운행관리자 → 관리자(배차담당)
- **교대 요청서**: 기사 → 운행관리자 승인
- **차량 정비 요청서**: 기사/관리자 → 관리자 → 대표
- **사고 보고서**: 기사 → 운행관리자 → 관리자 → 대표
- **물품 구매 요청서**: 관리자 → 대표
- **경비 청구서**: 기사/관리자 → 관리자 → 대표

#### 핵심 기능

- **결재선 관리**: 결재 양식별 기본 결재선 설정, 기안자가 결재선 수정 가능
- **대결 처리**: 결재자 부재 시 위임된 대결자가 대리 결재
- **합의/협의**: 타 부서 의견 요청 기능
- **열람**: 결재 완료 문서의 열람 권한 설정
- **문서번호 자동 생성**: 연도-부서-일련번호 형식
- **알림**: 결재 요청/승인/반려 시 푸시 알림 발송

#### 급여 프로세스

```
급여 기준일 설정 → 근태 데이터 집계 → 수당 자동 계산 →

급여 이체 → 급여명세서 개별 발송(앱 푸시)
```

| 출퇴근 기록 | GPS 기반 또는 QR코드/NFC 방식 | 차고지 도착 시 자동 출근 인식 |
| 연차 관리 | 연차 발생/사용/잔여 자동 계산 | 배차표 연동 자동 반영 |
| 병가 관리 | 병가 신청/승인/증빙 관리 | 대체 기사 자동 배정 연동 |
| 초과근무 | 초과근무 시간 자동 집계 | 운행시간 기반 자동 계산 |
| 교대근무 | 교대 패턴 설정/관리 | 5일 근무/2일 휴무 사이클 관리 |
| 휴가 캘린더 | 팀/전사 휴가 현황 조회 | 배차 가능 인원 실시간 파악 |

#### 근태 데이터 소스

- 모바일 앱 GPS 기반 출퇴근 인증
- 차량 OBD/DTG 연동 (운행 시작/종료 자동 기록)
- 지문인식/안면인식 단말기 (차고지 설치)
- 관리자 수동 입력 (예외 처리)

### 3.4 사내 게시판/공지사항/소통 채널

#### 게시판 유형

| 게시판 | 작성 권한 | 조회 권한 | 용도 |
|--------|----------|----------|------|
| 공지사항 | ADMIN 이상 | 전체 | 회사 공지, 정책 변경, 중요 안내 |
| 안전 게시판 | MANAGER 이상 | 전체 | 안전 수칙, 사고 사례, 교육 자료 |
| 자유게시판 | 전체 | 전체 | 기사 간 소통, 건의사항 |
| 노선 게시판 | ADMIN 이상 | 전체 | 노선별 운행 정보, 도로 상황 |
| 정비 게시판 | 전체 | 전체 | 차량 상태, 정비 요청/완료 |
| 인사 게시판 | ADMIN 이상 | 전체 | 인사 발령, 채용, 교육 |

#### 실시간 소통 기능

- **그룹 채팅**: 노선별/팀별 채팅방
- **1:1 메시지**: 관리자-기사 간 직접 소통
- **긴급 공지**: 전체 기사에게 즉시 전달 (돌발 상황, 기상 악화 등)
- **설문/투표**: 근무 만족도 조사, 교육 일정 투표

### 3.5 배차/스케줄 관리 (버스 회사 특화)

#### 배차표 자동 생성 로직

```
입력: 노선 목록, 기사 목록, 근무 패턴(5일/2일), 제약 조건
  ↓
[AI 기반 최적 배차 알고리즘]
  - 기사별 근무일/휴무일 자동 할당
  - 연속 근무일 제한 (최대 6일)
  - 최소 휴식 시간 보장 (8시간 이상)
  - 기사 선호도/숙련도 반영
  - 공평한 주말/공휴일 배분
  ↓
출력: 월간 배차표 (기사-날짜-노선-시간 매핑)
```

#### 실시간 배차 관리

- **당일 슬롯 드랍**: 기사 갑작스런 결근 시 빈 슬롯 발생
- **대체 기사 자동 추천**: 해당 시간 쉬는 기사 목록 자동 필터링
- **푸시 알림 발송**: 대체 가능 기사들에게 알림 → 수락/거절
- **실시간 배차 현황 보드**: 관리자가 한눈에 당일 배차 상태 확인

#### 월간/주간/일간 뷰

| 뷰 | 대상 | 표시 정보 |
|----|------|----------|
| 월간 뷰 | 관리자 | 전체 기사 월간 배차표 (달력 형태) |
| 주간 뷰 | 관리자/기사 | 주간 상세 스케줄 |
| 일간 뷰 | 관리자 | 당일 전체 배차 현황 (실시간) |
| 내 스케줄 | 기사 | 본인 배차 정보만 표시 |

| 항목 | 세부 내용 |

| 등록 정보 | 등록일, 차령, 폐차예정일 (차령 제한 관리) |
| 보험 정보 | 보험종류, 보험사, 만기일, 보장내용 |
| 검사 정보 | 정기검사일, 다음 검사 예정일, 배출가스 검사 |
| 정비 이력 | 정비일자, 정비항목, 비용, 정비업체 |

#### 차량 정비 관리

```
정기 점검 스케줄 설정 → D-7 알림 → 정비 요청 생성 →
정비 시행 → 결과 기록 → 다음 점검일 자동 갱신
```

- **타이어 관리**: 마모 상태, 교체 이력, 재생타이어 사용 여부
- **소모품 관리**: 엔진오일, 브레이크 패드 등 교체 주기 자동 알림
- **긴급 정비**: 운행 중 고장 시 즉시 보고 → 대체 차량 배정

#### 차령 관리

여객자동차 운수사업법에 따라 차량 종류/사업 종류별 차령 제한이 있으며, 안전성 요건 충족 시 최대 2년 연장 가능. 시스템에서 차령 만료 D-90, D-30, D-7 자동 알림 필요.

### 3.7 안전 관리

#### 사고 보고 워크플로우

```
사고 발생 → 기사 즉시 보고(앱)
  - 사고 유형 선택 (접촉, 추돌, 전복, 화재 등)
  - 사진/동영상 첨부
  - GPS 위치 자동 기록
  - 관련자 정보 입력

```

#### 안전 교육 관리

- **법정 교육**: 반기 1회 시/도 주관 안전교육 이수 관리
- **자체 교육**: 월 1회 회사별 자체 안전교육 실시/기록
- **교육 이수 현황**: 기사별 교육 이수 현황 대시보드
- **미이수자 알림**: 교육 미이수 기사 자동 알림

#### 운행기록(DTG) 분석

운행기록장치에서 수집하는 데이터:
- 순간속도, 분당 엔진회전수(RPM), 브레이크 신호
- GPS 좌표, 방위각, 가속도

분석 항목:
- **과속 횟수/구간**: 제한속도 초과 감지
- **급가속/급감속**: 위험 운전 패턴 분석
- **급핸들(급차로변경)**: 위험 운전 감지
- **운전 습관 점수**: 기사별 안전 운전 점수 산출
- **운전 습관 트렌드**: 월별/분기별 추이 분석

## 4. 락인(Lock-in) 전략

### 4.1 사내 시스템 교체가 어려운 이유

| 전환 비용 유형 | 설명 | 비중 |
|-------------|------|------|
| **데이터 마이그레이션** | 수년간 축적된 운행/인사/급여 데이터 이전 | 매우 높음 |
| **학습 비용** | 새 시스템에 대한 전 직원 재교육 (특히 고령 기사) | 높음 |
| **프로세스 재설정** | 결재선, 업무 규칙, 자동화 설정 재구성 | 높음 |
| **연동 재구축** | BIS/BMS, DTG, 교통카드 등 외부 시스템 재연동 | 매우 높음 |
| **다운타임 리스크** | 전환 기간 중 업무 공백 발생 위험 | 중간 |
| **심리적 저항** | 익숙한 시스템에 대한 선호 (현상 유지 편향) | 중간 |

> Flexera 2023 보고서에 따르면, 전체 기업의 47%가 데이터 마이그레이션을 공급업체 전환의 가장 큰 장벽으로 꼽았다.

### 4.2 성공적인 B2B SaaS 락인 사례

#### Slack

- **전략**: 사용자당 과금 → 조직 전체 도입 유도 → 커뮤니케이션 히스토리 축적
- **결과**: 직원 80% 이상이 Slack을 사용하는 조직은 경쟁 플랫폼으로의 전환 가능성이 62% 낮음
- **핵심**: 소통 데이터의 축적이 곧 전환 비용

#### Salesforce

- **전략**: 깊은 커스터마이제이션(맞춤 필드, 워크플로우, 앱) + 에코시스템(AppExchange) 의존성 생성
- **결과**: 한 번 도입하면 탈출이 극도로 어려움. 수년간 맞춤 설정한 CRM 데이터와 프로세스를 다른 시스템으로 옮기는 비용이 막대
- **핵심**: 커스터마이제이션 깊이 = 전환 비용

#### SAP

- **전략**: 회사의 모든 프로세스(회계, 인사, 물류, 생산)를 하나로 통합
- **결과**: SAP 교체 프로젝트는 수 년, 수십억 원 규모. 전문 인력(SAP 컨설턴트) 없이는 운영 불가
- **핵심**: 통합 범위의 넓이 = 전환 비용

### 4.3 버스 업계 특화 락인 포인트

#### 1단계: 필수 기능으로 진입

```
배차 관리 시스템 (핵심 Pain Point 해결)
  → 기존 엑셀/수기 배차에서 전환
  → 즉각적인 가치 제공
```

#### 2단계: 데이터 축적으로 가치 증가

```
운행 데이터 + 기사 근태 + 배차 히스토리 축적
  → 과거 데이터 기반 AI 최적 배차 제안
  → 데이터가 쌓일수록 시스템의 가치 증가 (데이터 네트워크 효과)
  → 이 데이터를 다른 시스템으로 옮기는 것은 사실상 불가
```

#### 3단계: 모든 업무 통합으로 의존성 극대화

```
배차 + 급여 + 결재 + 안전 + 차량 + 소통 = 올인원
  → 회사의 모든 업무 프로세스가 플랫폼에 의존
  → 한 모듈만 교체하는 것도 불가 (데이터 연동 깨짐)
  → SAP 수준의 통합 락인
```

#### 4단계: 법규 준수 자동화로 전환 부담 극대화

```
DTG 연동 + 적성검사 관리 + 안전교육 이수 + 사고보고
  → 법적 의무사항 관리가 시스템에 의존
  → 전환 시 법규 위반 리스크 발생
  → 전환 자체가 "위험한 결정"이 됨
```

#### 핵심 락인 매트릭스

| 락인 요소 | Slack형 (소통) | Salesforce형 (커스텀) | SAP형 (통합) |
|----------|:----------:|:-------------:|:---------:|
| 채팅/소통 히스토리 | O | | |
| 배차 규칙/AI 학습 데이터 | | O | |
| 결재 양식/결재선 설정 | | O | |
| 급여 체계/수당 규칙 | | O | |
| 운행 데이터 축적 | | | O |
| 차량/정비 이력 | | | O |
| 기사 인사 기록 | | | O |
| 법규 준수 자동화 | | | O |

## 5. 기술적 구현 고려사항

### 5.1 멀티테넌시 설계

여러 버스 회사를 하나의 플랫폼에서 서비스하기 위한 멀티테넌시 설계가 필수.

#### 데이터 격리 모델 비교

| 모델 | 설명 | 장점 | 단점 | 적합 시나리오 |
|------|------|------|------|-------------|
| **DB 단위 분리** | 테넌트별 별도 DB | 완벽한 격리, 성능 독립 | 관리 복잡, 비용 높음 | 대규모 고객, 규제 준수 필요 |
| **스키마 단위 분리** | 하나의 DB, 테넌트별 스키마 | 적절한 격리, 관리 용이 | 스키마 마이그레이션 복잡 | 중간 규모 |
| **행 단위 격리(RLS)** | 동일 테이블, tenant_id로 구분 | 단순, 비용 효율적 | 격리 수준 낮음, 쿼리 실수 위험 | 초기/소규모 |

#### 권장 전략: 행 단위 격리(RLS)로 시작 → 성장 시 하이브리드로 진화

```sql
-- PostgreSQL Row Level Security 예시
ALTER TABLE schedules ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON schedules
  USING (tenant_id = current_setting('app.current_tenant')::uuid);

-- 애플리케이션에서 테넌트 설정
SET app.current_tenant = 'company-uuid-here';
```

### 5.2 SSO (Single Sign-On) / 통합 인증

#### 구현 방식

| 방식 | 프로토콜 | 용도 |
|------|---------|------|
| **SAML 2.0** | XML 기반 | 기업 내부 시스템 간 인증 (대기업 고객 대응) |
| **OAuth 2.0 + OIDC** | JSON/JWT 기반 | 소셜 로그인, 모바일 앱 인증 |
| **자체 JWT** | JWT | 자체 웹/앱 간 인증 (초기 단계) |

#### 구현 모델

1. **Delegation Model (인증 대행)**: 대상 서비스의 인증 방식을 변경하지 않고, Sign On Agent가 사용자 인증 정보를 관리하여 대신 로그인
2. **Propagation Model (인증 정보 전달)**: 통합 인증 서버에서 인증 토큰을 발급받아 전달, 각 서비스에서 토큰으로 사용자 인식

#### 권장 구현

```
[초기] 자체 JWT 기반 인증 (웹 + 모바일 통합)
  ↓
[성장기] OAuth 2.0 + OIDC 도입 (SSO 기능 확장)
  ↓
[엔터프라이즈] SAML 2.0 추가 지원 (대기업 고객 요구 대응)
```

#### 교통카드 시스템

- 승하차 결제 시스템과 연동하여 승객 수 파악
- 결제정보, 버스 위치, 운행정보 등을 관제센터로 송신
- BMS/BIS 연동 단말기가 차량에 설치되어 데이터 수집

#### BIS(버스정보시스템) 구조

```
[차량 단말기(GPS + 교통카드)] → [통신망(LTE)]
       ↓
[BMS 서버 (운수회사/지자체)]
  - 노선 관리
  - 기사 근무 교대 관리
  - 차량 운행 관리
       ↓
[BIS 서버 (지자체)]
  - 실시간 차량 위치
  - 도착 예정 시간 예측
  - 승객 정보 제공
       ↓
[정류소 안내 단말기 / 모바일 앱 / 웹]
```

#### 연동 고려사항

- 지자체별 BIS API가 다를 수 있음 (표준화 미비)
- 실시간 위치 데이터 오차 발생 가능 (배차 간격 긴 노선)
- 교통카드 데이터는 지자체/카드사 API를 통해 접근

| 결재 | 종이 문서, 구두 | 추적 불가, 증빙 관리 어려움 |
| 차량 관리 | 정비 수첩, 엑셀 | 이력 누락, 예방 정비 미비 |
| 안전 관리 | 수기 보고서 | 체계적 분석 불가 |
| 법규 관리 | 담당자 기억/수첩 | 만료일 놓침, 과태료 리스크 |

#### 경쟁 우위 기회

```
[일반 그룹웨어]     [운수업 특화 시스템]      [우리 제품]
  메일/결재/근태     배차/BIS/DTG            올인원 통합
  HR/급여            차량/안전               + AI 배차
  게시판/소통        법규 준수               + 모바일 앱
                                           + 데이터 분석
  ─────────────── × ────────────────── = ──────────────
  업종 특화 없음      그룹웨어 기능 없음      모든 것을 하나에
```

### 7.3 차별화 포인트

| 차별화 요소 | 상세 내용 | 경쟁 우위 |
|-----------|----------|----------|
| **AI 자동 배차** | Claude API 기반 최적 배차 생성, 자연어 규칙 입력 | 엑셀/수기 대비 압도적 효율 |
| **올인원 통합** | 배차+HR+결재+소통+차량+안전 통합 | 여러 도구 사용 대비 통합성 |
| **기사 전용 모바일 앱** | 배차 확인, 출퇴근, 요청, 알림 | 카카오톡 업무 사용 대체 |
| **법규 자동 관리** | 자격/검사/교육 만료 자동 알림 | 과태료/사고 리스크 감소 |
| **데이터 기반 인사이트** | 안전 점수, 운행 효율, 비용 분석 | 경영 의사결정 지원 |
| **SaaS 모델** | 초기 비용 낮음, 즉시 사용 가능 | 자체 구축 대비 도입 용이 |

### 7.4 글로벌 시장 동향

버스 배차 관리 시스템 소프트웨어 글로벌 시장:
- 2024년 기준 **24억 달러** 규모
- 2025~2034년 **연평균 10.6% 성장** 전망
- 성장 동력: 스마트 시티, 전기 버스 전환, 실시간 데이터 분석 수요 증가

## 8. 제품 전략 제언

### 8.1 MVP → 성장 로드맵

#### Phase 1: MVP (현재 ~ 3개월)
**핵심 가치**: "엑셀 배차를 AI로 자동화"

- [x] AI 기반 월간 배차표 자동 생성
- [x] 기사 모바일 앱 (배차 확인, 휴무 요청)
- [x] 관리자 웹 (배차 관리, 기사 관리)
- [x] 당일 슬롯 드랍 → 푸시 알림 → 수락
- [x] Excel 내보내기

#### Phase 2: 핵심 확장 (3 ~ 6개월)
**핵심 가치**: "배차 + HR 통합"

- [ ] 전자결재 시스템 (휴무, 교대, 경비)
- [ ] 근태관리 (GPS 출퇴근, 연차/병가)
- [ ] 급여 관리 기본 (수당 계산, 명세서)
- [ ] 사내 게시판/공지사항
- [ ] 차량 기본 정보 관리

#### Phase 3: 올인원 (6 ~ 12개월)
**핵심 가치**: "회사 전체를 하나의 플랫폼에서"

- [ ] 급여 관리 고도화 (4대보험, 원천세, 이체)
- [ ] 차량 관리 고도화 (정비 이력, 보험, 검사)
- [ ] 안전 관리 (사고 보고, DTG 연동, 안전 점수)
- [ ] 자격/교육 관리 (만료 자동 알림)
- [ ] 민원 관리
- [ ] 경영 대시보드 (통계/리포트)

#### Phase 4: 플랫폼화 (12개월 ~)
**핵심 가치**: "버스 업계 표준 플랫폼"

- [ ] 멀티테넌시 (여러 회사 서비스)
- [ ] BIS/BMS/DTG 연동
- [ ] 교통카드 데이터 연동
- [ ] API 마켓플레이스 (서드파티 연동)
- [ ] SSO / SAML 지원
- [ ] 법규 업데이트 자동 반영

### 8.2 가격 전략

| 플랜 | 대상 | 기능 | 가격 (예시) |
|------|------|------|-----------|
| **스타터** | 소규모 (30명 이하) | 배차+근태+게시판 | 월 15만원 |
| **프로** | 중규모 (31~100명) | 올인원 전체 기능 | 월 40만원 |
| **엔터프라이즈** | 대규모 (100명+) | 프로 + DTG연동 + API + 전담 지원 | 별도 협의 |

### 8.3 GTM(Go-To-Market) 전략

1. **초기 타깃**: 인천 지역 중소 버스 회사 (성민버스 등)
2. **확산**: 인천 → 경기 → 수도권 → 전국
3. **영업 전략**: 엑셀 배차의 고통을 해결하는 메시지 중심
4. **무료 체험**: 1개월 무료 → 데이터 축적 후 유료 전환 (자연스러운 락인)
5. **레퍼런스**: 초기 고객 성공 사례를 마케팅 자산화

### 8.4 핵심 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 배차 생성 시간 절감율 | 80% 이상 | 수기 vs 자동 비교 |
| 기사 앱 일일 활성 사용률 | 90% 이상 | DAU/전체 기사 |
| 결재 처리 시간 | 평균 4시간 이내 | 기안~최종승인 시간 |
| 고객사 이탈률 | 연 5% 미만 | 월별 추적 |
| 기사 만족도 | 4.0/5.0 이상 | 분기 설문 |
